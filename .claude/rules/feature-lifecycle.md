# Feature Development Lifecycle

## Protocol

Every new feature MUST follow the 4-phase lifecycle:

```
/feature [name]
  Phase 0: PRE-FLIGHT → verify required skills exist
  Phase 1: PLAN       → sparc-prd-mini → docs/features/<n>/sparc/
  Phase 2: VALIDATE   → requirements-validator (swarm, max 3 iterations)
  Phase 3: IMPLEMENT  → swarm of agents + parallel tasks
  Phase 4: REVIEW     → brutal-honesty-review (swarm)
```

## Rules

### Planning (Phase 1)
- ALL features get SPARC documentation, no exceptions
- Documentation lives in `docs/features/<feature-name>/sparc/`
- sparc-prd-mini runs Gate to assess task clarity before starting
- Use sparc-prd-mini in MANUAL mode for complex features, AUTO for minor
- sparc-prd-mini delegates to explore, goap-research, problem-solver-enhanced via view()
- Architecture.md MUST be consistent with project's root Architecture
- Commit docs before implementation

### SPARC Completeness Gate (between Phase 1 and Phase 2) — HARD GATE

**BLOCKS Phase 2 (and therefore Phase 3) until all mandatory documents exist.**

After Phase 1 completes, verify the following 5 mandatory documents exist in `docs/features/<feature-name>/sparc/`:

```
REQUIRED (all 5 must exist — BLOCKS if any missing):
  ✅ PRD.md            — Product requirements, user stories
  ✅ Specification.md   — API contracts, acceptance criteria, validation rules
  ✅ Pseudocode.md      — Algorithms, data flow, state machines
  ✅ Architecture.md    — Component diagrams, DB schema, dependencies
  ✅ Refinement.md      — Edge cases, testing strategy, security considerations

OPTIONAL (generated by sparc-prd-mini but not blocking):
  ⚠️ Solution_Strategy.md
  ⚠️ Research_Findings.md
  ⚠️ Completion.md
  ⚠️ Final_Summary.md
```

**Enforcement logic:**
```
AFTER Phase 1 commit:
  missing = []
  FOR doc IN [PRD.md, Specification.md, Pseudocode.md, Architecture.md, Refinement.md]:
    IF NOT exists docs/features/<feature-name>/sparc/{doc}:
      missing.append(doc)

  IF missing is not empty:
    WARN "⛔ SPARC Completeness Gate FAILED — missing: {missing}"
    GENERATE missing documents from project context + existing SPARC docs
    COMMIT "docs(feature): complete SPARC docs for <feature-name>"
    RE-CHECK (must pass before proceeding)

  LOG "✅ SPARC Completeness Gate PASSED — 5/5 mandatory docs present"
```

**Rationale (INS-002):** Documentation drives code generation. Incomplete docs lead to hallucinated implementations. This gate ensures code is ALWAYS generated from complete specifications, not partial context.

### Validation (Phase 2) — HARD GATE
- Run requirements-validator as swarm (parallel validation agents)
- Minimum score: 70/100 average, no BLOCKED items
- Fix gaps in docs, not in code
- Max 3 iterations — if not passing, escalate to user
- Commit validation-report.md
- **BLOCKS Phase 3** until validation passes

### Implementation (Phase 3)
- Read SPARC docs — don't hallucinate code
- Modular design — components reusable across projects
- Use Task tool for parallel work on independent modules
- Commit after each logical change (not at end)
- Run tests in parallel with development
- Format: `feat(<feature-name>): <description>`

### Review (Phase 4)
- Use brutal-honesty-review with swarm of agents
- No sugar-coating — find real problems
- Fix all critical and major issues before marking complete
- Benchmark performance after implementation
- Commit review-report.md

## When to Skip Phases

| Scenario | Skip | Justification |
|----------|------|---------------|
| Hotfix (1-5 lines) | Phase 1-2 | Too small for full SPARC |
| Config change | Phase 1-2 | No new functionality |
| Dependency update | Phase 1-2 | No new design needed |
| Refactoring | Phase 1 only | Validate + implement + review |
| New feature | NEVER skip | Full lifecycle always |

For skipped phases, still run Phase 4 (brutal-honesty-review) on the changes.

## Artifact Verification (INS-007 compliance)

After each feature completion, the following MUST exist:
- `docs/features/<feature-name>/sparc/PRD.md` — product requirements
- `docs/features/<feature-name>/sparc/Specification.md` — API contracts, acceptance criteria
- `docs/features/<feature-name>/sparc/Pseudocode.md` — algorithms
- `docs/features/<feature-name>/sparc/Architecture.md` — component design, DB schema
- `docs/features/<feature-name>/sparc/Refinement.md` — edge cases, testing strategy
- `docs/features/<feature-name>/sparc/validation-report.md` — validation results
- `docs/features/<feature-name>/review-report.md` — code review results

Minimum 5 SPARC docs + validation-report + review-report = 7 artifacts per feature.
Missing artifacts trigger a warning. `/run` audit step checks coverage matrix.
